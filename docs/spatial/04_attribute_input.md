## 1. 가. 주요속성값 — 컬럼 목록

원문 순번(①~) 기준. 타입은 전부 CHAR.

| 원문 번호 | 컬럼명(컬럼ID) | 길이 | 입력값 |
|---|---|---|---|
| ① | 기본키(PRIMARY KEY) | 41 | ③레이어ID + `_` + ⑩일련번호 |
| ② | 참조키(FOREIGN KEY) | 38 | 3차원객체코드 (`M_...`) |
| ③ | 레이어(LAYER)ID | 35 | 레이어코드 (`L_...`) |
| ④ | 건물ID | 17 | UFID(공간객체등록번호) |
| ⑤ | 층(FLOOR) | 6 | 지하1층 → `B01B01`, 지상1층 → `F01F01`, 로비층 → `L01L01`, 지하1층지상1층 → `B01F01` |
| ⑥ | 구분(DIVISION) | 2 | 이동 `MV`, 안내 `IF`, 편의 `CV`, 운수 `TP`, 안전 `SF`, 소방 `FF`, 판매 `SL`, 의료 `MD`, 주거 `HS`, 업무 `BS`, 관리 `MN`, 구조 `IS` |
| ⑦ | 구조(CLASSIFY) | 2 | **공간 `RM`**, 시설물 `FC`, 천정 `CL`, 벽 `WL`, 바닥 `FL`, 문 `DR`, 창문 `WD`, 기둥 `PL` (8종) |
| ⑧ | 세부항목(KIND) | 50 | "카페", "식당", "소화전", "공중전화" 등 |
| ⑨ | 세부항목_영문(KANDENG) | 100 | "CAFE", "RESTAURANT" 등 |
| ⑩ | 일련번호(SERIAL) | 5 | "00001", "00002" |
| ⑩* | 명칭(NAME) | 50 | "스타벅스", "김밥천국". **시설물의 경우 현재층_공간(+_시설명) — "1층_복도_소화전"**. 객석/좌석일 경우 객석번호 입력 |
| ⑫ | 재질(QAL) | 6 | §2.1 참조 |
| ⑬ | 경사(SLP) | 6 | §2.2 참조 |
| ⑭ | 진행방향(DIR) | 6 | §2.3 참조 |
| ⑮ | 내외부(SKT) | 6 | §2.4 참조 |
| ⑯ | 교통약자(HANDICAP) | 6 | §2.5 참조 |
| ⑰ | 첫번째점(FPOINT) | 36 | 경로연결 시작 지점 PRIMARY KEY 값 |
| ⑱ | 두번째점(SPOINT) | 36 | 경로연결 끝 지점 PRIMARY KEY 값 |
| ⑯* | 기타 | 50 | (입력값 정의 없음) |

> **원문 번호 오탈자 주석**: 원문은 ⑩(일련번호)과 ⑩(명칭), ⑯(교통약자)과 ⑯(기타)에 같은 번호를 중복 부여하고 ⑪은 건너뛴다(위 표에서 `*`로 표시). 기본키 정의 "③+`_`+⑩"의 ⑩은 문맥상 **일련번호(SERIAL)**를 가리킨다. 또한 ⑨의 컬럼ID는 "KANDENG"으로 표기되어 있으나 나. 레이어정의 테이블에서는 "KINDENG"이므로 KINDENG이 올바른 것으로 판단한다.

### 기본키 형식

```
PRIMARY KEY (CHAR 41) = 레이어코드 (CHAR 35) + "_" + 일련번호 (CHAR 5)
예: L_B00100000001AULW1_F01F01_BS_RM_00001
```

> **원문 길이 표기 불일치 주석**: 원문의 CHAR 35(레이어코드)/CHAR 41(기본키)은 원문 자신의 예시 UFID 17자와 맞지 않는다 — `L_`(2)+UFID(17)+`_`(1)+FLOOR(6)+`_`(1)+DIVISION(2)+`_`(1)+CLASSIFY(2) = 레이어코드 32자, 기본키 32+1+5 = **38자**(위 예시도 38자). CHAR 35/41은 UFID 20자를 전제한 표기로 추정된다. POC는 UFID 17자 mock을 사용하므로 실제 기본키 길이는 38자다(`naming.ts` doc comment와 상호 참조).

POC의 `SpatialSpace` id(spaceId)가 이 형식을 채택한다.

## 2. 속성 코드값 (CHAR 6)

### 2.1 재질 (QAL)

| 코드 | 의미 |
|---|---|
| `QAL000` | 미분류 |
| `QAL001` | 방화 |
| `QAL002` | 유리 |
| `QAL003` | 철재 |
| `QAL004` | 나무 |
| `QAL005` | 콘크리트 |
| `QAL006` | 종이 |
| `QAL007` | 플라스틱 |
| `QAL999` | 기타 |

> **원문 오탈자 주석**: 원문은 방화를 `QALQ001`(7자)로 표기하나, 컬럼 길이 CHAR(6) 및 나머지 코드의 6자리 체계(`QAL` + 3자리)와 불일치하는 오탈자로 판단하여 **`QAL001`로 정규화**한다. 또한 원문의 콘크리트 표기 `"QAL005`는 닫는 따옴표가 누락된 단순 오식이다.

## 3. 나. 레이어정의 테이블 형식

원문은 4종의 레이어정의서 예시를 제공한다. 공통적으로 레이어 설명은 "시설명(건물명)+층+레이어명" 형식(예: 강릉아이스아레나_1층_경로).

### 3.1 경로네트워크 — 경로(선형) / 지점(점형)

- 레이어ID 예: `L_B00100000001AULW1_F01F01_MV_LINE` (경로, 선형) / `L_B00100000001AULW1_F01F01_MV_POINT` (지점, 점형)
- 컬럼 15종 (모두 CHAR, NotNull): PRIMARYKEY(41, KEY), LAYER(35), ID=UFID(17), FLOOR(6), DIVISION(2), KIND(50), KINDENG(100), SERIAL(5), NAME(50), SLP(6), DIR(6), SKT(6), HANDICAP(6), FPOINT(36), SPOINT(36)
- CLASSIFY 컬럼이 없고 레이어코드 4번째 자리에 `LINE`/`POINT`를 사용 — 별표 5 CLASSIFY 코드 체계 밖의 경로 전용 확장.
- QAL(재질) 컬럼이 없고 대신 FPOINT/SPOINT(경로 연결)와 HANDICAP을 가진다.

### 3.2 공간(면형) / 시설물(점형)

- 레이어ID 예: `L_B00100000001AULW1_F01F01_MV_RM` (공간, 면형 — 레이어명 "이동_공간") / `L_B00100000001AULW1_F01F01_MV_FC` (시설물, 점형 — 레이어명 "이동_시설물")
- 컬럼 15종 (모두 CHAR, NotNull): PRIMARYKEY(41, KEY), FOREIGNKEY=3차원객체코드(38), LAYER(35), ID=UFID(17), FLOOR(6), DIVISION(2), CLASSIFY(2), KIND(50), KINDENG(100), SERIAL(5), NAME(50), QAL(6, DEFAULT `QAL000`), SLP(6, DEFAULT `SLP000`), DIR(6, DEFAULT `DIR000`), SKT(6, DEFAULT `SKT000`)
- HANDICAP/FPOINT/SPOINT 컬럼이 없고, QAL~SKT 4종은 DEFAULT 미분류(…000) 값을 가진다.

## 4. POC 매핑 관련 메모

- `SpatialSpace`는 §3.2 공간(면형) 레이어의 컬럼(기본키/레이어코드/UFID/FLOOR/DIVISION/CLASSIFY=RM/KIND/KINDENG/SERIAL/NAME/QAL/SLP/DIR/SKT)을 채택하고, HANDICAP은 안전약자 대피 시나리오 대비로 추가 채택한다.
- `SpatialFacility`의 name은 ⑩* 명칭 규칙("현재층_공간_시설명", 예: `1층_복도_소화전`)을 따른다.
- 경로네트워크(LINE/POINT, FPOINT/SPOINT)는 Phase 7 토폴로지(webbuilder 내비 메시) 연동 범위로 이연한다 → [06_schema_mapping.md](./06_schema_mapping.md) 참조.
